Why Firebase?

Firebase is Google's app development platform, and BlogAI uses two of its core services: Authentication and Cloud Firestore.

Authentication handles everything related to user accounts — signing up, logging in, password resets, and Google sign-in. Instead of building all of that from scratch (which is complex and easy to get wrong from a security standpoint), Firebase gives you a battle-tested system that handles password hashing, session tokens, and OAuth flows. When a user clicks "Sign in with Google" on your app, Firebase is doing all the heavy lifting behind the scenes.

Cloud Firestore is a NoSQL database that stores your app's data — user profiles, saved blog posts, subscription status, and more. It's a "serverless" database, meaning you don't need to set up or maintain a database server. Your app reads and writes data directly, and Firestore handles scaling, backups, and availability. It also has real-time sync, so if data changes, your app can update instantly without refreshing the page.

Client SDK vs Admin SDK — What's the Difference?

Firebase gives you two sets of keys because your app talks to Firebase from two different places, each with different trust levels:

The Client SDK keys run in the user's browser. They're used for things like signing in, signing up, and reading data that the user is allowed to see. These keys are "public" in the sense that anyone can see them in your website's source code — and that's fine by design. Firebase security rules control what each user can actually access, not the keys themselves.

The Admin SDK keys (the service account) run on your server only. They have full, unrestricted access to your Firebase project — they can read any data, delete any user, bypass all security rules. This is why the private key must be kept secret in .env.local and never exposed to the browser. Your server uses these keys for trusted operations like verifying authentication tokens in middleware and managing user records.

Why OpenRouter?

The core feature of BlogAI is generating blog posts using AI. When a user types a topic and clicks "Generate," the app needs to send that request to an AI model and stream the response back. But AI models like Claude don't run on your computer — they run on powerful servers operated by companies like Anthropic. To use them, your app makes API calls over the internet.

OpenRouter is an API gateway that sits between your app and AI model providers. Instead of signing up directly with Anthropic, OpenAI, Google, and others, you get a single OpenRouter API key that gives you access to many models. BlogAI uses it to call Anthropic's Claude Sonnet model, which is excellent at writing long-form content like blog posts. OpenRouter handles billing, rate limiting, and failover — you just make one API call and get back the generated text. New accounts come with free credits, so you can start generating right away without entering payment details.

Why Stripe?

BlogAI is a SaaS (Software as a Service) app, which means users pay a monthly subscription to access premium features. Stripe is the payment platform that makes this possible. It handles everything you'd never want to build yourself: securely collecting credit card numbers, processing payments, managing recurring subscriptions, handling failed payments, issuing refunds, and staying compliant with financial regulations.

You'll set up three things in Stripe: API keys (so your app can talk to Stripe's servers), a product and price (the "Pro Plan" that users subscribe to), and a webhook (so Stripe can notify your app when something happens, like a successful payment or a cancelled subscription). Stripe has a "Test mode" that lets you simulate all of this with fake credit cards — no real money changes hands during development. The test card number 4242 4242 4242 4242 always succeeds, which is how you'll verify everything works.

Stripe API Keys guided

Stripe gives you two keys. The publishable key is used in the browser to create checkout sessions — it's safe to expose publicly. The secret key runs on your server and can charge cards, create subscriptions, and access sensitive data — it must stay private in .env.local.

  1. Go to Stripe Dashboard — make sure Test mode is ON
  2. Go to Developers > API keys
  3. Copy the Publishable key and Secret key, paste them to Claude Code

Stripe Product guided

In Stripe, a Product represents what you're selling (e.g., "Pro Plan"), and a Price defines how much it costs and how often (e.g., $19/month). Your app uses the Price ID to tell Stripe exactly what to charge when a user clicks "Subscribe." You create these in the Stripe Dashboard, not in code — this means you can change pricing, add new plans, or run promotions without touching your codebase.

  1. In Stripe Dashboard, go to Product catalog > + Add product
  2. Name it "Pro Plan", set pricing to Recurring at $19/month (to match the pricing on your landing page)
  3. Save, then copy the Price ID (price_...) and paste to Claude Code
ValueEnv Variable
Price IDSTRIPE_PRO_PRICE_ID

Stripe Webhook (Local Testing) guided

When a user completes a payment, cancels their subscription, or updates their card, Stripe needs a way to tell your app about it. It does this through webhooks — HTTP requests that Stripe sends to a URL on your server whenever something important happens. Your app listens at /api/webhooks/stripe and updates the user's subscription status in Firestore accordingly.

During local development, your computer isn't publicly accessible on the internet, so Stripe can't reach it directly. The Stripe CLI solves this by creating a secure tunnel — it listens for events on Stripe's servers and forwards them to your local localhost:3000. The webhook signing secret lets your app verify that incoming webhook requests are genuinely from Stripe and haven't been tampered with.

  1. Open a separate terminal (keep the current one available)
  2. Log in to Stripe CLI: 
    stripe login
  3. Start the webhook listener: 
    stripe listen --forward-to localhost:3000/api/webhooks/stripe
  4. Copy the webhook signing secret (whsec_...) and paste to Claude Code
ValueEnv Variable
Signing secretSTRIPE_WEBHOOK_SECRET
⚠
Keep that terminal running while testing payments locally. The webhook listener forwards Stripe events to your dev server.
💡
Tip: If stripe login or stripe listen gives you an error, copy the entire error message and paste it into Claude Code. It can usually diagnose the issue immediately — common fixes include installing the Stripe CLI or restarting your terminal.

Deploy to Production

Your app works locally — now let's put it on the internet so anyone can use it.

Why Vercel?

Vercel is the company that created Next.js, and their hosting platform is purpose-built for it. When you deploy to Vercel, it automatically detects your Next.js project, builds it, and hosts it on a global CDN (Content Delivery Network) — meaning your app loads fast for users anywhere in the world. It also handles HTTPS certificates, custom domains, and automatic deployments when you push to GitHub.

The free tier is generous enough for learning and small projects. You don't need to configure servers, manage Docker containers, or worry about scaling. Push your code, and Vercel takes care of the rest. This is what "serverless deployment" looks like in practice.

Connect GitHub to Vercel guided

  1. Go to vercel.com and click "Sign Up"
  2. Choose "Continue with GitHub" — this links your GitHub account so Vercel can access your repositories
  3. Click "Add New..." > "Project"
  4. Find your saas-boilerplate repository in the list and click "Import"
ⓘ
If you don't see your repository, click "Adjust GitHub App Permissions" to grant Vercel access to it.

Production Stripe Webhook guided

Locally, the Stripe CLI forwarded webhook events to your computer. In production, Stripe needs to send events directly to your Vercel URL. You'll create a webhook endpoint in the Stripe Dashboard that points to your live app.

  1. Go to Stripe Dashboard > Developers > Webhooks
  2. Click "Add endpoint"
  3. Set the endpoint URL to your Vercel app's webhook route: 
    https://your-project.vercel.app/api/webhooks/stripe
  4. Under "Select events to listen to", add these three events: 
    checkout.session.completed
customer.subscription.updated
customer.subscription.deleted
  5. Click "Add endpoint" to save
  6. On the endpoint page, click "Reveal signing secret" — copy this new whsec_... value
  7. Go to Vercel > your project > Settings > Environment Variables and update STRIPE_WEBHOOK_SECRET with the new production secret
  8. Redeploy the app so it picks up the new webhook secret
ⓘ
Your local webhook secret (from stripe listen) and your production webhook secret (from the Stripe Dashboard) are different. The local one only works with the Stripe CLI tunnel. The production one only works with your live Vercel URL.

Switch Stripe from Test Mode to Live Mode guided

Throughout this setup you've been using Stripe in Test mode — a sandbox where no real money changes hands. Test mode keys start with pk_test_ and sk_test_, and only the fake card number 4242 4242 4242 4242 works. This is perfect for development, but when you're ready to accept real payments from real customers, you need to switch to Live mode.

Live mode keys start with pk_live_ and sk_live_ and connect to real payment networks. When a customer enters their credit card on your live site, Stripe will actually charge them. You'll also need to create a new webhook endpoint for live mode (or update your existing one) because test and live mode are completely separate environments in Stripe — they have different keys, different webhook secrets, and different transaction histories.

⚠
You can skip this step while learning. It's perfectly fine to keep your deployed app in test mode until you're ready to launch for real. Everything will work the same — just no real charges. Come back to this step when you're ready to go live.
  1. Go to Stripe Dashboard
  2. Toggle "Test mode" OFF in the top-right corner (it will switch to live mode)
  3. Stripe may require you to activate your account first. This involves:
    • Providing your business details (or selecting "Individual")
    • Adding a bank account for payouts
    • Verifying your identity
  4. Once activated, go to Developers > API keys (now showing live keys)
  5. Copy your live Publishable key (pk_live_...) and live Secret key (sk_live_...)
  6. Go to Product catalog — you'll need to create your Pro Plan product again in live mode (test products don't carry over). Set it to $19/month recurring and copy the new Price ID
  7. Go to Developers > Webhooks > Add endpoint in live mode: 
    https://your-project.vercel.app/api/webhooks/stripe
    Add the same three events: checkout.session.completed, customer.subscription.updated, customer.subscription.deleted
  8. Copy the live webhook signing secret (whsec_...)
  9. Update these four values in Vercel > Settings > Environment Variables:
    VariableNew Value
    NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEYYour pk_live_... key
    STRIPE_SECRET_KEYYour sk_live_... key
    STRIPE_PRO_PRICE_IDYour new live price_... ID
    STRIPE_WEBHOOK_SECRETYour live whsec_... secret
  10. Redeploy the app in Vercel
  11. Test with a real card to confirm everything works (you can refund yourself immediately from the Stripe Dashboard)
ⓘ
Keep your local .env.local on test keys. Your local development environment should always use test mode so you don't accidentally process real charges while building features. Only your Vercel production environment should have live keys.

Update Firebase Authorized Domains guided

Firebase Authentication only allows sign-ins from domains you've explicitly approved. Your localhost is approved by default, but your new Vercel URL isn't — so Google sign-in and other auth flows will fail on your live site until you add it.

  1. Go to Firebase Console > Authentication > Settings tab
  2. Under "Authorized domains", click "Add domain"
  3. Add your Vercel domain (without https://): 
    your-project.vercel.app
  4. Click "Add"

Custom Domain (Optional) guided

Your app works at your-project.vercel.app, but if you own a domain (e.g., myblogai.com), you can connect it. Vercel handles the SSL certificate automatically.

  1. In Vercel, go to your project > Settings > Domains
  2. Type your domain name and click "Add"
  3. Vercel will show you DNS records to add at your domain registrar (e.g., Namecheap, GoDaddy, Cloudflare)
  4. After DNS propagates (can take a few minutes to a few hours), your custom domain will be live
  5. Update NEXT_PUBLIC_APP_URL in Vercel to your custom domain and redeploy
  6. Add the custom domain to Firebase's authorized domains list (same as above)

Troubleshooting

Common issues and how to fix them.

💡
Tip: For any error you encounter, the fastest fix is usually to copy the entire error message and paste it into Claude Code. It has context about this project's setup and can diagnose most issues immediately. Don't worry about understanding the error yourself — just paste it in and let Claude Code figure it out.
pnpm: command not found
Run npm install -g pnpm, then close and reopen your terminal.
git: command not found
Install Git from git-scm.com, then restart VS Code completely.
node: command not found
Install Node.js LTS from nodejs.org, then restart VS Code completely.
ERR_PNPM_NO_IMPORTER_MANIFEST_FOUND
You're in the wrong folder. Make sure you cd into the folder that contains package.json.
Blank page or console errors
Check .env.local — look for any empty values. Every variable needs to be filled in.
Auth not working (can't sign up or log in)
Make sure Email/Password and Google sign-in are enabled in Firebase Console under Authentication > Sign-in method.
Firestore permission denied
Make sure you created the Firestore database in test mode. You can check this in Firebase Console > Firestore Database > Rules.
FIREBASE_ADMIN_PRIVATE_KEY errors
Make sure the value is wrapped in double quotes in .env.local and has \n for newlines. Example: FIREBASE_ADMIN_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\nMIIE...\n-----END PRIVATE KEY-----\n"
"OpenRouter API key is not configured"
Add OPENROUTER_API_KEY to .env.local and restart the dev server (pnpm dev).
Blog generation fails or returns errors
Check that your OpenRouter API key is correct and your account has credits. Visit openrouter.ai/keys to verify.
Stripe checkout not redirecting
Verify all 4 Stripe environment variables are filled in: NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY, STRIPE_SECRET_KEY, STRIPE_PRO_PRICE_ID, and STRIPE_WEBHOOK_SECRET.
Port 3000 already in use
Run pnpm dev -- -p 3001 to use a different port, then open http://localhost:3001 instead.
Vercel build fails
Check the build logs in Vercel > Deployments > click the failed deployment. Common causes: missing environment variables (check that every variable from .env.local is added in Vercel Settings > Environment Variables) or TypeScript errors in your code.
Auth works locally but not on Vercel
Your Vercel domain needs to be added to Firebase's authorized domains. Go to Firebase Console > Authentication > Settings > Authorized domains > Add domain. Add your your-project.vercel.app domain (without https://).
Stripe payments work locally but not in production
You need a separate production webhook endpoint in Stripe. Go to Stripe Dashboard > Developers > Webhooks > Add endpoint pointing to https://your-app.vercel.app/api/webhooks/stripe. Copy the new signing secret and update STRIPE_WEBHOOK_SECRET in Vercel's environment variables, then redeploy.
NEXT_PUBLIC_APP_URL is wrong after deploying
Update this value in Vercel > Settings > Environment Variables to match your actual Vercel URL (e.g., https://your-project.vercel.app), then redeploy. This URL is used for Stripe redirect URLs and other absolute links.
